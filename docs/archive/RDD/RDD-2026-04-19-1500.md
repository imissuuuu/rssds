# RDD - 2026-04-19-1500
**Project Name:** Phase 7 Stage 2 — 実画像描画

## 1. 機能一覧

### 機能要件
- **FR-1**: `Article::imageUrls`（Stage 1 完了済）をオンデマンドで取得・デコード・描画。Stage 1 のテキスト URL リスト表示は撤去
- **FR-2**: ダウンロード/デコード/リサイズ/swizzle前処理は worker thread、`C3D_TexInit` + `C3D_TexLoadImage` は render thread で実行
- **FR-3**: スケーリング先は **256×256 POT 上限**（RGBA8 = 256 KiB/枚）。元画像はアスペクト比維持で 256 内に収める
- **FR-4**: 同時保持テクスチャ数は **可視範囲 ± 1 枚** をオンデマンド管理。範囲外は `C3D_TexDelete` で解放
- **FR-5**: テクスチャ配置は `linearAlloc`（`onVram=false`）を標準採用。VRAM bank 偏り問題を回避
- **FR-6**: ダウンロード / デコード / 確保失敗時は `[画像取得失敗]` プレースホルダを表示
- **FR-7**: 1画像あたり最大 **2 MiB**（HEAD で Content-Length 確認、超過時 GET 中断）
- **FR-8**: 1記事あたり最大 10 枚（Stage 1 仕様継承）

### 非機能要件
- **NFR-1**: Old 3DS で動作。VRAM 予算（3 MiB × 2 bank）を圧迫しない
- **NFR-2**: 記事遷移時に全画像テクスチャを解放（リーク無し）
- **NFR-3**: Worker thread の処理中も UI スクロール応答を維持
- **NFR-4**: Worker stack 初期値 **64 KiB**（ScummVM 3DS 実例）。不足時は拡張
- **NFR-5**: Render thread での `C3D_TexInit` + `C3D_TexLoadImage` は **1 フレーム 1 枚まで**（fps 維持）
- **NFR-6**: 画像取得失敗・デコード失敗・サイズ超過が起きても他の画像や本文表示に影響しない

## 2. 詳細仕様

### 2.1 ダウンロード（worker thread）
- **入力**: 画像 URL（`Article::imageUrls` から）
- **処理**:
  1. HEAD で Content-Length を確認、2 MiB 超ならスキップ
  2. HEAD 不可なら GET で受信しつつサイズ監視、超過時中断
  3. メモリバッファに保持
- **出力**: `std::vector<uint8_t>` または失敗フラグ

### 2.2 デコード（worker thread）
- **入力**: 画像バイナリバッファ
- **処理**: `stbi_load_from_memory()` で RGBA8 デコード
- **出力**: 元幅・元高さ・RGBA ピクセル配列（top-left 起点）

### 2.3 リサイズ（worker thread）
- **入力**: RGBA 配列、元サイズ
- **処理**: 長辺を 256 に合わせアスペクト比維持で縮小（バイリニア）。元が 256 以下なら原寸
- **出力**: 縮小後 RGBA 配列（`imgW`, `imgH`、いずれも ≤ 256）

### 2.4 Swizzle 前処理（worker thread）
- **入力**: 縮小後 RGBA 配列
- **処理**:
  1. POT texture サイズ決定: `texW = next_pot(imgW)`, `texH = next_pot(imgH)`（最小 8、最大 256）
  2. `texW * texH * 4` byte の出力バッファを `linearAlloc`（worker thread から確保可、後で render thread が `C3D_TexLoadImage` の src として使用）
  3. 各 texel について `dstOffset = (((y>>3)*(texW>>3) + (x>>3))*64 + morton8(x&7,y&7))*4`、書き込み順は **A, B, G, R**
  4. パディング領域は 0 埋め（透明）
- **出力**: swizzled buffer（render thread 受け渡し用）

### 2.5 テクスチャ作成・upload（render thread）
- **入力**: swizzled buffer + `texW`/`texH`/`imgW`/`imgH`
- **処理**:
  1. `C3D_TexInitWithParams`（`format=GPU_RGBA8`, `onVram=false`）。失敗時は失敗フラグ
  2. `C3D_TexLoadImage`（`face=GPU_TEXFACE_2D`, `level=0`） で swizzled buffer を upload
  3. `Tex3DS_SubTexture` を作成: `width=imgW`, `height=imgH`, `left=0`, `right=imgW/texW`, `top=1.0f`, `bottom=1.0f - imgH/texH`
  4. `C2D_Image` を組み立てて cache へ登録
  5. swizzled buffer を `linearFree`
- **制約**: 1 フレームあたり最大 1 枚まで処理（NFR-5）

### 2.6 描画（render thread）
- ArticleView の本文末尾に画像セクションを描画
- 各画像を縦に並べる（マージン数 px）
- スクロール時に画像 Y 座標を計算に含める

### 2.7 オンデマンド管理（render thread）
- **入力**: スクロール位置、画像メタ（取得後の表示サイズ）
- **処理**:
  1. 表示位置から画像ごとの矩形を計算
  2. 可視範囲 ± 1 枚の URL set を算出
  3. cache 内の不要 URL は `C3D_TexDelete` + `linearFree` で解放
  4. 未取得の必要 URL は worker キューへ投入

### 2.8 ライフサイクル
- **記事選択時**: cache クリア → 画像セクションのレイアウト枠決定（メタは未取得）→ スクロールに応じて 2.7 が動作
- **記事遷移時**: worker キュー drain → cache 全解放
- **アプリ終了時**: worker thread 停止 → cache 全解放

## 4. 追加設計情報

### 4.1 実装上の制約（PDF リサーチ準拠）
- **POT 必須**: 8〜1024 の 2 の冪のみ。non-POT は POT に padding して `Tex3DS_SubTexture` で切る
- **Swizzle**: 8×8 タイル Morton 順 + texel byte 順は A,B,G,R（`C2D_Color32` とは別物）
- **SubTexture 向き**: `top > bottom` で非回転（top-left 起点画像と整合）
- **Threading**:
  - libctru allocator (`vramAlloc`/`linearAlloc`/`linearFree`) には内部同期が見当たらない → 同一 thread から呼ぶか自前直列化
  - `C3D_SyncTextureCopy` / `C3D_TexLoadImage` は citro3d の共有 frame/queue state を触る → render thread で実行
  - 安全な分担: worker = download/decode/resize/swizzle前処理、render = `C3D_TexInit*` + `C3D_TexLoadImage`
- **VRAM**: 3 MiB × 2 bank、bank 跨ぎ不可。`vramSpaceFree()` の総和では成否保証できない。本実装は `linearAlloc` 標準で回避
- **Stack**: `threadCreate` の `stack_size` は明示必須（デフォルトなし）

### 4.2 既存コードへの影響（概要のみ、詳細は DESIGN）
- `source/image_loader.{cpp,h}` 新規 — worker thread 管理、download/decode/resize/swizzle
