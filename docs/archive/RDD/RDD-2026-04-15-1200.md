## 2. 詳細仕様

### 抽出パイプライン
入力: HTML 文字列（UTF-8、ネットワーク取得済み）
出力: 本文プレーンテキスト（改行保持）

**Phase A: pre-clean**
- 入力 HTML を lexbor でパース
- `<script>` `<style>` `<nav>` `<header>` `<footer>` `<aside>` `<form>` `<iframe>` を DOM から削除
- ノイズパターン（CSS クラス / id のブラックリスト）に一致するノードを削除

**Phase B: Readability 本体**
- 各 `<p>` `<div>` 等の候補ノードにスコアを付与
  - CJK 文字数ベース（現行の Miniflux 流バイト数ではない）
  - リンク密度ペナルティ、ノイズクラス名ペナルティ
  - 親ノードへのスコア伝播
- 最高スコアのノードを本文候補として選定
- 兄弟ノードの本文マージ（スコア閾値以上のもの）

**Phase C: post-clean**
- 本文候補サブツリー内の残存ノイズを除去（広告/共有ボタン等のクラス名パターン）
- テキスト化して改行整形

### 進捗コールバック
- シグネチャ例: `void (*progress_cb)(int percent, const char* stage, void* user)`
- 呼び出しポイント: パーサ開始 / pre-clean 完了 / スコアリング完了 / post-clean 完了 / 終了
- user ポインタは呼び出し側で任意のコンテキスト渡しに利用

### ノイズパターン
- ハードコード配列。エントリ形式: `{ pattern_type, pattern_string }`
- `pattern_type`: CLASS_CONTAINS / ID_EQUALS / TAG_NAME 等
- 対象サイト観察で 10〜30 個を手動選定

### ベースライン計測
- 実装開始前に現行 `html_strip.cpp` で主要 3 サイト × 各 3 記事の抽出時間を計測
- 結果を HANDOFF 側に記録し、新実装の比較基準とする

## 3. UI/UX考慮事項
本フェーズは抽出エンジンの裏方置換が中心。UI 変更はなし。

進捗コールバックは埋め込みのみで、UI 配線は別フェーズ（プログレスバー実装時）に行う。

## 4. 追加設計情報

### 実装上の制約
- lexbor は既に `lib/lexbor/` にあり `source/html_strip.cpp` で使用中（Makefile の SOURCES/INCLUDES に登録済）。追加ライブラリ導入不要
- lexbor の DOM API は `lxb_dom_node_t` のツリー走査が中心。親子関係・クラス名・タグ名取得で Readability の要件は満たせる
- lexbor ノード上の文字列は `mraw` 管理下。`lexbor_free()` ではなく document destroy で一括解放

### 既存コードへの影響
- `source/html_strip.cpp` — 抽出ロジック全面書き換え（外部公開 API は維持）
- `source/html_strip.h` — 進捗コールバック引数の追加
- `source/article.cpp` — 抽出関数呼び出し側で進捗コールバックを受け取る（UI には流さない、空実装可）
- テスト用サンプル HTML 保存先を新規に確保（場所は DESIGN で決定）

### 確定事項サマリ
- 現行 Miniflux 相当 → Readability.js 相当へアルゴリズム置換（完全置換）
- 3 フェーズ構成、ノイズパターンはハードコード
- 進捗は実計測ベースのコールバック。UI は別フェーズ
- 性能目標は平均 5〜6 秒、事前にベースライン取得
