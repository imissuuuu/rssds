# RDD - 2026-04-13-0001
**Project Name:** ワークフローエンジンのデータドリブン化

## 1. 機能一覧

### 機能要件
- **FR-1**: ワークフロー定義を `.claude/harness-config.json` で管理する
- **FR-2**: フロー名・ステップ数・ステップ名・メッセージを config で定義可能にする
- **FR-3**: デフォルト config に現行 full / light を含めて配布する
- **FR-4**: `commit-dev.sh` のブランチ制限を `commit.branch` で制御する（デフォルト：制限なし）
- **FR-5**: `install.sh` は config が既存なら上書きしない
- **FR-6**: README.md に config 仕様・ハーネス設計意図を記載。`wf help` コマンドで README パスを案内する

### 非機能要件
- **NFR-1**: 追加依存なし（jq のみ）
- **NFR-2**: 既存プロジェクトは `install.sh update` で config が追加される

## 2. 詳細仕様

### FR-1 / FR-2: harness-config.json

ワークフローエンジン（wf.sh）が読み込む設定ファイル。

**入力**: `.claude/harness-config.json`（JSON形式）
**処理**: `wf.sh` が `jq` で読み込み、フロー定義を取得
**出力**: ステップ遷移の状態管理（既存の `workflow_state.json` と同等）

```json
{
  "workflows": {
    "full": {
      "steps": [
        { "name": "Requirement Definition", "message": "run /requirement-analyst" },
        { "name": "System Design", "message": "run /system-design" },
        { "name": "Implementation", "message": "read coding-patterns.md, implement per DESIGN" },
        { "name": "Build & Test", "message": "run /build-and-test; append new patterns to coding-patterns.md" },
        { "name": "Review", "message": "run /reviewer -> fix criticals" },
        { "name": "Commit", "message": "run /handoff -> /commit" }
      ]
    },
    "light": {
      "steps": [
        { "name": "Intent Confirmation", "message": "agree what/why/how; read coding-patterns.md" },
        { "name": "Implementation", "message": "implement per agreed design" },
        { "name": "Build & Test", "message": "run /build-and-test; append new patterns to coding-patterns.md" },
        { "name": "Review", "message": "run /reviewer -> fix criticals" },
        { "name": "Commit", "message": "run /handoff -> /commit" }
      ]
    }
  },
  "commit": {
    "branch": ""
  }
}
```

- `workflows`: キーがフロー名、値がステップ配列。フロー数・ステップ数は自由
- `steps[].name`: ステップ表示名（`wf status` で出力）
- `steps[].message`: ステップの指示テキスト（`wf status` で出力、skill名含む）
- `commit.branch`: 許可するブランチ名。空文字 = 制限なし

### FR-3: デフォルト配布

`install.sh` の初回インストール時に上記デフォルト値を含む `harness-config.json` を `.claude/` に配置する。

### FR-4: ブランチ制限

**入力**: `harness-config.json` の `commit.branch` 値
**処理**: `commit-dev.sh` が config を読み、値が非空なら現在ブランチと照合
**出力**: 一致 → コミット実行、不一致 → エラー、空 → チェックスキップ

### FR-5: update 時の保護

`install.sh update` で `harness-config.json` は：
- 存在しない → 新規作成（デフォルト値）
- 存在する → 上書きしない（スキップ）

### FR-6: ドキュメント・ヘルプ

- README.md にハーネスの設計思想と config 仕様を記載
- `wf help` サブコマンドを追加。README のパスを出力する

## 3. UI/UX考慮事項

CLI出力のみ。`wf status` の出力フォーマット変更：

**変更前（ハードコード）:**
```
[wf] full/task step 1/6: Requirement Definition -> next: System Design
```

**変更後（config駆動、フォーマットは同等）:**
```
[wf] full/task step 1/6: Requirement Definition -> next: System Design
```

表示上の変化はない。内部のデータソースが変わるだけ。

## 4. 追加設計情報

### 影響ファイル
- `.claude/scripts/wf.sh` — ハードコード配列を config 読み込みに置換
- `.claude/scripts/commit-dev.sh` — ブランチチェックを config 参照に変更
- `.claude/harness-config.json` — 新規作成
- `install.sh` — config ファイルのインストール/保護ロジック追加
- `README.md` — ハーネス説明・config 仕様を記載
- `.claude/skills/commit/SKILL.md` — dev固定の記述を config 参照に修正

### 制約
- jq が必須（既存と同じ）
- config の JSON バリデーションは最小限（jq 読み込み失敗時のエラーメッセージで十分）
